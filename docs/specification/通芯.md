# 通芯（Grid）作成仕様書

- **Version**: 1.0
- **Target**: AI coding agents
- **Language**: 日本語
- **Project**: IFC Web-CAD / BIM CAD

---

## 1. 目的

本仕様は、IFC Web-CAD における **通芯（Grid）** の作成・管理・表示・命名ルールを定義する。

通芯は、壁・柱・梁・設備などの配置基準となる **建物全体の参照要素** であり、通常の実体要素（Wall / Column / Beam など）とは異なる。

**重要原則:**

> 通芯は「レベル専属の要素」ではなく、建物全体で共有される参照系として扱う。

---

## 2. 基本方針

### 2.1 通芯の本質

通芯は以下の役割を持つ。

- 柱の配置基準
- 梁の配置基準
- 壁の位置合わせ基準
- 寸法・図面表現の基準
- BIM モデル全体の座標参照

### 2.2 実体要素との違い

通芯は Wall / Column のような物理要素ではなく、**Reference Element** として扱う。

- **Grid** = 参照要素
- **Wall / Column / Beam** = 実体要素

---

## 3. レベルとの関係

### 3.1 結論

通芯は原則として **全レベル共通** のものを使用する。

**理由**

- 実務上、通芯は建物全体の基準線である
- 柱・梁は複数階で同じ通芯に整列することが多い
- レベルごとに別 Grid を持つと整合性が崩れやすい
- IFC/BIM 的にも Grid は建物参照要素として扱う方が自然

### 3.2 ただし表示はレベルごとに制御可能

通芯そのものは共通でも、**どのビュー・どのレベルで表示するか** は制御できるようにする。

**ルール**

- Grid 定義は building 共通
- 表示設定は view / level ごと
- 必要なら「このレベルでは非表示」にできる
- 必要なら通芯の表示範囲（2D extent / 3D extent）を持つ

### 3.3 非標準ケース

一部フロアだけ異なる通芯系が必要なケースもある。

**例:**

- 増築部分
- 塔屋
- 別棟接続部
- 特殊フロアだけの補助通芯

この場合は、原則の共通通芯とは別に **補助通芯系（Auxiliary Grid Set）** を追加可能とする。

**ルール**

- 主通芯系: building 共通
- 補助通芯系: 特定範囲・特定用途向け
- ただし、レベルごとに完全独立の通芯を大量に持つ設計は避ける

---

## 4. 通芯の配置単位

通芯は次の階層で保持する。

```text
Project
 └ Site
    └ Building
       ├ Levels
       ├ Grids
       └ Elements
```

**重要**

Grid は **Level 配下ではなく、Building 配下** に置く。これは Tree UI でも同じ。

---

## 5. データモデル

### 5.1 GridLine

```ts
interface GridLine {
  id: string;
  name: string;
  curve: GridCurve;
  kind: "Primary" | "Auxiliary";
  visible: boolean;
  locked: boolean;
  bubbleStart?: boolean;
  bubbleEnd?: boolean;
  extents?: GridExtents;
}
```

### 5.2 GridCurve

MVP では直線のみ必須。将来円弧対応可。

```ts
type GridCurve =
  | {
      type: "Line";
      start: Vec3;
      end: Vec3;
    }
  | {
      type: "Arc";
      center: Vec3;
      radius: number;
      startAngle: number;
      endAngle: number;
    };
```

### 5.3 GridExtents

表示範囲制御。

```ts
interface GridExtents {
  mode: "Model" | "ViewSpecific";
  minZ?: number;
  maxZ?: number;
}
```

## 6. 作成単位

通芯の作成方法は以下をサポートする。

### 6.1 単独作成

1. Grid ツール起動
2. 2 点指定
3. 通芯作成
4. 名称自動付与

### 6.2 連続作成

1. 基準通芯を作成
2. オフセット距離を指定
3. 複数本を一括作成
4. 名称を連番付与

### 6.3 配列作成

1. 開始位置
2. ピッチ
3. 本数
4. 方向
5. 一括生成

これは実務で非常に重要。

## 7. 命名仕様

### 7.1 結論

名称は **作成時に自動付与** する。その後、ユーザーが任意に編集可能とする。

**理由**

- 作成直後に無名だと管理しにくい
- 柱・梁・壁スナップ時に識別しづらい
- Tree / Properties / 寸法注記で名前が必要
- 実務上、通芯は最初から名前で扱う

### 7.2 基本命名ルール

通芯名称は方向ごとに系列を持つ。

**典型例:**

- X 方向通芯: `1, 2, 3, 4, ...`
- Y 方向通芯: `A, B, C, D, ...`

**推奨ルール**

- 一方の方向は数字系列
- 他方の方向は英字系列

ただし、どちらを数字・英字にするかはプロジェクト設定で変更可能とする。

### 7.3 命名タイミング

**必須ルール**

- 単独作成時: 確定直後に自動採番
- 連続作成時: 一括で連番付与
- 配列作成時: 生成順に自動命名

**例**

- 最初の縦通芯 → `A`
- 次の縦通芯 → `B`
- 最初の横通芯 → `1`
- 次の横通芯 → `2`

### 7.4 ユーザー編集

作成後、ユーザーは名前を編集できる。

**ルール**

- 重複名は原則警告
- ただし許可するか禁止するかは設定可能
- BIM/CAD としては重複禁止推奨

### 7.5 自動命名のための系列管理

```ts
interface GridNamingState {
  xSeries: GridSeries;
  ySeries: GridSeries;
}

interface GridSeries {
  scheme: "Numeric" | "Alphabetic";
  nextValue: string;
}
```

**例**

- `xSeries = Numeric` → `1, 2, 3...`
- `ySeries = Alphabetic` → `A, B, C...`

## 8. 方向判定と系列割当

通芯作成時に、その線がどちらの系列に属するかを判定する。

### 8.1 基本ルール

- ほぼ縦方向の線 → Alphabetic 系列
- ほぼ横方向の線 → Numeric 系列

### 8.2 判定方法

線分方向ベクトル `d = end - start`

```text
if |dx| >= |dy|:
    horizontal-like
else:
    vertical-like
```

### 8.3 注意

これはあくまで自動候補であり、ユーザーが系列を変更できる余地を持たせる。

## 9. 表示仕様

### 9.1 3D 表示

- 通芯は参照線として表示
- バブル（名称表示）を端部に表示可能
- 3D では薄色・点線でもよい

### 9.2 2D / Plan 表示

- 通芯線を明確に表示
- バブル名称を端部に表示
- アクティブ level で強調表示可能

### 9.3 レベルごとの表示

Grid 自体は共通だが、以下の表示設定を持てるようにする。

```ts
interface GridViewOverride {
  gridId: string;
  viewId: string;
  visible: boolean;
}
```

MVP では view override までは不要でもよい。まずは building 共通表示 + level filter 程度で十分。

## 10. スナップ仕様

通芯は強い snap 対象である。

### 10.1 対応 snap

- Grid line snap
- Grid intersection snap
- Bubble endpoint snap（任意）
- Parallel/perpendicular reference

### 10.2 用途

- 柱を通芯交点に置く
- 梁端点を通芯交点へ合わせる
- 壁を通芯に沿わせる
- 寸法の基準にする

### 10.3 優先度

推奨 snap 優先度:

1. Grid intersection
2. Existing element endpoint
3. Existing element center
4. Grid line
5. Free cursor point

---

## 11. Tree UI 仕様

通芯は Tree で以下のように表示する。

```text
Project
 └ Building
    ├ Levels
    ├ Grids
    │   ├ A
    │   ├ B
    │   ├ C
    │   ├ 1
    │   ├ 2
    │   └ 3
    └ Elements
```

**重要**

- Grid は Level 配下に置かない
- Building 配下の独立カテゴリとする

## 12. Grid の編集仕様

### 12.1 編集可能項目

- 名前
- 端点
- 方向
- 表示
- ロック
- Bubble start/end
- 系列種別（必要なら）

### 12.2 移動

通芯移動時は、通常は **参照要素のみ移動** する。実体要素は自動追従しない。

**理由**

- BIM/CAD では Grid は参照系
- Grid を動かしたから壁や柱を強制的に動かすと危険

将来、必要なら「追従移動」コマンドを別途持つ。

---

## 13. 削除仕様

通芯削除は可能だが、要素との関係は参照情報として扱う。

### 13.1 削除時の挙動

- Grid を削除しても柱・梁・壁自体は削除しない
- ただし、その Grid を参照していた配置補助情報は消える
- 必要なら警告を出す

---

## 14. Auto Dimension / Auto Layout との関係

将来、通芯は以下に使える。

- 自動寸法
- 柱の自動配置
- 梁の自動生成
- フレーム構造生成
- 通り芯名称ベースの要素位置表現

そのため、最初から **安定した名前と一意 ID** を持たせること。

---

## 15. IFC 整合

IFC と整合するため、Grid は参照要素として保持する。

**原則**

- Grid は building / spatial structure に紐づく参照系
- Wall / Column / Beam とは分離する
- 名前・曲線・表示範囲を保持する

MVP では IFC export まで未実装でも、内部モデルはそれに合わせる。

---

## 16. 補助通芯（Auxiliary Grid）仕様

特殊用途のため、補助通芯を許可する。

### 16.1 用途

- 一部フロアのみの補助基準
- 設備芯
- 外構基準
- 改修用仮設基準

### 16.2 ルール

- `kind = Auxiliary`
- 主通芯とは別カテゴリで表示可能
- 必要なら特定 view のみ表示

### 16.3 重要

補助通芯があるからといって、レベルごとに Grid を完全分離する設計にはしないこと。

---

## 17. 推奨デフォルト動作

### 17.1 作成時

- 作成時に自動命名
- 方向から系列を自動判定
- Building 配下に追加
- 全レベル共通参照として登録

### 17.2 表示時

- すべての level で参照可能
- active level では強調表示
- view ごとの表示 ON/OFF は将来拡張

### 17.3 編集時

- 名前変更可能
- 重複時は警告
- 移動しても実体要素は自動追従しない

---

## 18. AI への実装指示

AI は以下を守って実装すること。

### 必須

- Grid は Building 配下で管理する
- Grid は Level 配下に置かない
- Grid は参照要素として扱う
- 名称は作成時に自動付与する
- 方向ごとに命名系列を分ける
- Grid line snap と Grid intersection snap を実装する

### 推奨

- 連続作成
- 配列作成
- 補助通芯 kind
- Bubble 表示
- 重複名警告

### 禁止

- レベルごとに独立した主通芯系を大量に持つ設計
- Grid を実体要素として扱うこと
- Grid 移動時に実体要素を自動で動かすこと
