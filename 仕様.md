# IFC Web-CAD Full AI Build Specification
Version: 1.0
Language: TypeScript
Framework: Next.js
Rendering: WebGPU
Goal: Build a browser-based IFC-compatible BIM CAD application using an element-based architecture.

===============================================================================
0. PRIMARY OBJECTIVE
===============================================================================

Build a web-based BIM CAD system with the following properties:

- Runs in the browser
- Uses WebGPU for rendering
- Uses element-based BIM modeling
- Supports IFC-compatible internal shape hierarchy
- Supports wall / column / slab / door / window / space editing
- Avoids direct mesh editing
- Uses procedural geometry generation
- Supports undo / redo
- Supports future IFC import/export extension

The single most important architectural rule is:

Element → Topology → Geometry → Mesh → Rendering

Meshes are generated data only.
Meshes must never be treated as the primary editable model.

===============================================================================
1. MANDATORY ARCHITECTURAL RULES
===============================================================================

The implementation MUST follow these rules:

1. Element-centered modeling
2. Geometry kernel isolation
3. Meshes are derived, not authored
4. IFC-compatible shape hierarchy
5. Incremental rebuild using dirty flags
6. BIM relationships are handled in a topology layer
7. UI actions must go through command objects
8. Rendering must be separated from BIM model logic
9. Geometry kernel must contain no UI logic
10. Renderer must contain no BIM topology logic

===============================================================================
2. MVP SCOPE
===============================================================================

The MVP must implement:

- Viewer
- Camera controls
- Selection
- Wall creation
- Column creation
- Slab creation
- Door insertion into wall
- Window insertion into wall
- Wall joins (butt and miter)
- Properties panel
- Project tree
- Snap system
- Dirty rebuild system
- Space detection from wall boundaries
- Basic export-ready IFC-aligned internal model

The MVP does NOT need:

- Full IFC import
- Full IFC export serializer
- Collaboration
- Multi-user sync
- Advanced freeform modeling
- Full BRep kernel
- Fillet / chamfer on arbitrary solids
- NURBS editing
- Server-side heavy boolean processing

5. FILE RESPONSIBILITIES
===============================================================================

Each file must have a clear responsibility.

Example responsibilities:

- model/elements/WallElement.ts
  Defines the wall element data structure only.

- topology/joins/WallJoinResolver.ts
  Determines how walls connect.
  Must not contain rendering code.

- geometry/builders/WallGeometryBuilder.ts
  Creates geometric wall surfaces from wall parameters and topology results.

- mesh/builders/WallMeshBuilder.ts
  Converts wall geometry into triangle mesh data.

- renderer/core/Renderer.ts
  Submits meshes to WebGPU and draws them.

- ui/tools/WallTool.ts
  Handles user interaction for wall creation.
  Must not directly generate final mesh.
  It must create commands.

===============================================================================
6. CODING STYLE RULES
===============================================================================

Use these rules:

- TypeScript strict mode
- Small focused classes/functions
- No giant files
- Prefer pure functions in geometry layer
- Prefer immutable updates in model layer where practical
- Explicit interfaces for all data models
- Avoid hidden shared mutable state
- Avoid circular dependencies
- Avoid renderer references in model layer
- Avoid UI references in geometry layer

Naming rules:

- Data model classes: WallElement, ColumnElement
- Builders: WallGeometryBuilder, SlabMeshBuilder
- Services / resolvers: WallJoinResolver, HostResolver
- Commands: CreateWallCommand, MoveElementCommand
- React UI files end in .tsx
- Utility pure functions use verb names

interface BaseElement {

===============================================================================
11. SHAPE SYSTEM
===============================================================================

Define shape hierarchy aligned with IFC concepts.

interface Shape {
  kind: ShapeKind
}

interface SweptSolidShape extends Shape {
  kind: "SweptSolid"
  profile: Profile
  height: number
  direction: Vec3
}

interface ClippingSolidShape extends Shape {
  kind: "ClippingSolid"
  base: SweptSolidShape
  modifiers: Modifier[]
}

interface BrepShape extends Shape {
  kind: "Brep"
  faces: unknown[]
}

interface MeshShape extends Shape {
  kind: "Mesh"
  vertices: Float32Array
  indices: Uint32Array
}

Rule:
Prefer SweptSolid first.
Then ClippingSolid.
Then Brep if needed.
Then Mesh as fallback.

===============================================================================
12. MODIFIER SYSTEM
===============================================================================

Special shapes must not immediately degrade into raw meshes.
Use modifiers.

interface Modifier {
  type: "Cut" | "Void" | "Notch" | "ClipPlane"
}

Examples:

interface NotchModifier extends Modifier {
  type: "Notch"
  width: number
  depth: number
  z0: number
  z1: number
  face: "X+" | "X-" | "Y+" | "Y-"
}

interface ClipPlaneModifier extends Modifier {
  type: "ClipPlane"
  origin: Vec3
  normal: Vec3
}

This allows a column with partial missing geometry to still remain a Column semantically.

===============================================================================
16. WALL GEOMETRY LOGIC
===============================================================================

Wall definition:
Axis line + thickness + height

Build process:

1. Read axis line
2. Compute left and right offset lines
3. Resolve joins at wall endpoints
4. Build wall footprint polygon
5. Apply openings
6. Extrude to wall height
7. Generate surfaces
8. Pass to mesh builder

Wall geometry should be procedural, not generic boolean-driven.

===============================================================================
17. COLUMN GEOMETRY LOGIC
===============================================================================

Column build process:

1. Read profile
2. Extrude to height
3. Apply modifiers if any
4. Generate surfaces
5. Pass to mesh builder

Support in MVP:

- Rectangle columns
- Circle columns
- Arbitrary closed profile columns
- Simple notch / cut modifiers

Do not implement full arbitrary freeform solids in MVP.

===============================================================================
18. SLAB GEOMETRY LOGIC
===============================================================================

Slab build process:

1. Read boundary polygon
2. Validate winding
3. Read optional holes
4. Triangulate top face
5. Build side faces
6. Build bottom face
7. Pass to mesh builder

===============================================================================
19. DOOR / WINDOW LOGIC
===============================================================================

Door and window are hosted elements.

Door/window insertion process:

1. Select host wall
2. Determine local position along wall axis
3. Store hostWallId
4. During wall geometry generation, split wall around opening
5. Rebuild wall surfaces
6. Build door/window own mesh separately

Avoid full 3D generic boolean subtraction for MVP.

Prefer segmented wall generation around the opening.

===============================================================================
20. SPACE DETECTION
===============================================================================

Space detection is a 2D planar graph problem.

Algorithm:

1. Project wall boundaries into 2D
2. Split edges at intersections
3. Build planar graph
4. Detect closed loops
5. Remove outer infinite region
6. Generate room polygons
7. Create SpaceElement objects

Use the user’s existing interest in polyline intersection / closed polygon extraction as inspiration.

27. DIRTY FLAG SYSTEM
===============================================================================

Each element can carry dirty flags.

Dirty categories:

- Parameters
- Topology
- Geometry
- Mesh
- Render

Update flow example:

Wall thickness changes
→ mark Parameters
→ topology maybe dirty if join affected
→ geometry rebuild
→ mesh rebuild
→ renderer update

Do not rebuild the entire model if one wall changes.
